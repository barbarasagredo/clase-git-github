Diferencia: Clonar vs Forkear

Clonar y forkear son formas de trabajar con repositorios. Imagina que clonar es como sacar una fotocopia de un libro para leerlo localmente, mientras que forkear es crear tu propia versión del libro en tu cuenta de GitHub.

Clonar un Repositorio

Clonar descarga una copia completa de un repositorio a tu computadora. Úsalo cuando quieras trabajar localmente, pero recuerda que no puedes subir cambios al repositorio original sin permisos. Ejemplo: clonas el código de un proyecto para probarlo. 

# Descarga el repositorio a tu máquina
git clone https://github.com/usuario/repositorio.git
Consejo: Verifica que tienes permisos si planeas enviar cambios al repositorio original.
Forkear un Repositorio

Forkear crea una copia del repositorio en tu cuenta de GitHub. Es ideal para contribuir a proyectos open source, ya que puedes editar tu copia y proponer cambios al original con un pull request. Ejemplo: forkeas un proyecto para añadir una funcionalidad. 

# Clona tu propio fork
git clone https://github.com/TU_USUARIO/repositorio-forkeado.git
Consejo: Mantén tu fork actualizado ejecutando git pull upstream main si el repositorio original se actualiza.
Estados de los Archivos en Git

Git organiza tus archivos según su estado, como un bibliotecario que registra qué libros han cambiado. Aprender estos estados te permitirá gestionar mejor tus proyectos.

  • Untracked: Archivos nuevos que Git no conoce, como un archivo que acabas de crear.
  • Tracked: Archivos que Git ya rastrea y monitorea para detectar cambios.
  • Modified: Archivos que has editado desde el último commit, pero aún no están listos para guardarse.
  • Staged: Archivos modificados que has marcado con git add para incluir en el próximo commit.
  • Deleted: Archivos eliminados que Git detecta; confirma su eliminación con git add o git rm. 
# Revisa el estado actual de tus archivos
git status
Consejo: Usa git status regularmente para mantenerte al tanto de los cambios en tu proyecto.
Tutoriales Paso a Paso

Sigue estas instrucciones detalladas para dominar las herramientas esenciales de control de versiones.

Una clave SSH actúa como una identificación segura para conectarte a GitHub. Te evita ingresar tu contraseña cada vez que subes o descargas código.

  1. Abre Git Bash o la terminal de tu sistema (puede ser Terminal en macOS o PowerShell en Windows).
  2. Genera tu clave SSH: Usa este comando, reemplazando tu-email@example.com con el correo asociado a tu cuenta de GitHub: 
    # Crea una clave SSH con el algoritmo ed25519
ssh-keygen -t ed25519 -C "tu-email@example.com"
    Atención: Si tu sistema no soporta ed25519, usa ssh-keygen -t rsa -b 4096 -C "tu-email@example.com".
  3. Confirma la ubicación: Presiona Enter para guardar la clave en ~/.ssh/id_ed25519. Puedes agregar una contraseña para mayor seguridad (opcional, pero recomendado).
  4. Copia la clave pública: Muestra tu clave pública con: 
    # Muestra la clave pública para copiarla
cat ~/.ssh/id_ed25519.pub
    Copia el texto que comienza con ssh-ed25519.
  5. Agrega la clave a GitHub: Ve a Configuración > SSH and GPG keys > New SSH key, pega la clave y asigna un nombre descriptivo (ej. "Mi Laptop").
  6. Prueba la conexión: Verifica que todo esté configurado correctamente: 
    # Prueba la conexión con GitHub
ssh -T git@github.com
    Si todo está bien, verás un mensaje como Hi username! You've successfully authenticated....

Consejo: Simplifica el uso de tu clave con ssh-agent: ejecuta eval "$(ssh-agent -s)" y ssh-add ~/.ssh/id_ed25519.
¡Cuidado! Nunca compartas tu clave privada (id_ed25519). Mantenla segura en tu computadora.

Configura tu nombre y correo en Git para que tus commits estén asociados a ti, como una firma digital. 

# Establece tu nombre de usuario
git config --global user.name "Tu Nombre"
# Establece tu correo electrónico
git config --global user.email "tu-email@example.com"
Consejo: Verifica tu configuración ejecutando git config --global --list.

Forkear crea una copia de un repositorio en tu cuenta de GitHub, perfecta para experimentar o contribuir a proyectos open source.

  1. Visita el repositorio original en GitHub.
  2. Haz clic en el botón Fork en la esquina superior derecha.
  3. Espera a que GitHub cree tu copia. ¡Listo! Ahora tienes tu propio repositorio.
Consejo: Usa forks para probar ideas sin afectar el proyecto principal.

Clona tu repositorio forkeado para editarlo en tu computadora. 

# Descarga tu fork a tu máquina
git clone https://github.com/TU_USUARIO/repositorio-forkeado.git
Consejo: Asegúrate de estar en el directorio correcto con cd ~/proyectos antes de clonar.

Crea un nuevo repositorio en GitHub para alojar tu código desde cero.

  1. Haz clic en New repository en tu perfil de GitHub.
  2. Elige un nombre único y agrega una descripción clara.
  3. Selecciona si será público o privado, y decide si incluir un README.
  4. Haz clic en Create repository.
Consejo: Incluye un archivo .gitignore al crear el repositorio para evitar subir archivos innecesarios.

Crea un repositorio local en tu computadora usando Visual Studio Code para versionar tu código.

  1. Abre tu proyecto en Visual Studio Code.
  2. Abre la terminal integrada (Ctrl + `).
  3. Inicializa el repositorio: git init
  4. Agrega todos los archivos: git add .
  5. Realiza el primer commit: git commit -m "Primer commit: inicialización del proyecto"
Consejo: Usa mensajes de commit descriptivos para documentar tus cambios.

Vincula tu repositorio local con uno en GitHub para sincronizar tus cambios. 

# Conecta tu repositorio local a GitHub
git remote add origin https://github.com/TU_USUARIO/TU_REPO.git
Consejo: Verifica la conexión con git remote -v.

Un commit guarda una instantánea de tus cambios. Usa mensajes claros para recordar qué hiciste. 

# Agrega todos los cambios
git add .
# Crea un commit con un mensaje claro
git commit -m "Cambios realizados en mi HTML y CSS"
Consejo: Usa git add nombre_archivo para agregar archivos específicos.

Envía tus commits locales a GitHub para compartirlos o respaldarlos. 

# Sube los cambios a la rama principal
git push -u origin main
Consejo: Si trabajas en una rama, usa su nombre en lugar de main (ej. feature/nueva-funcionalidad).
